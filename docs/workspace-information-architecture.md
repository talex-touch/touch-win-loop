# 工作台信息架构

更新时间：2026-04-03

本文描述当前 `touch-win-loop` 已落地的工作台对象关系，目标是让产品文档、接口设计和页面组织围绕同一套模型表达。

## 1. 核心对象

### Workspace

- 个人或 Team 级空间。
- 负责成员、席位、邀请、权限和 AI 配额边界。

### Project

- 工作推进主对象。
- 挂载项目设置、竞赛绑定、资源池、协作状态、AI 提案和 Issue。

### ProjectResource

- 项目内统一的一等资料对象。
- 文件、协作文档、画布都以资源形式存在。

### ProjectOutline

- 由项目设置和资源内容自动推导出的结构化导航。
- 用于浏览和定位，不承担编辑职责。

## 2. 当前页面对应关系

### 左侧资源管理器

- 资源列表
- 项目大纲
- 系统资料库导入
- 上传入口
- 协作资源新建入口

### 主工作区固定页签

- `仪表盘`
- `项目协作`
- `流程画布`
- `项目设置`

### 资源页签

- 某条资源被打开后，会以 `resource:<id>` 形式进入可关闭资源 tab

## 3. 资源模型

### 资源形态

| resourceKind | 说明 |
| --- | --- |
| `binary` | 上传文件或系统资料库引用 |
| `markdown` | 结构化协作文档 |
| `draw` | 画布类协作资源 |

### 协作用途

| collabPurpose | 适用形态 | 用户侧名称 |
| --- | --- | --- |
| `workflow` | `draw` | 流程画布 |
| `freeform` | `draw` | 自由画布 |
| `notes` | `markdown` | 协作文档 |

## 4. flow tab 的定位

当前约定中，`flow` 固定 tab 不再是旧的静态流程 checklist。

它的职责是：

1. 打开项目唯一的 `workflow` 画布
2. 渲染与资源预览页相同的真实画布协作视图
3. 保持“流程入口”与“workflow 资源”指向同一个底层对象

因此：

- `流程画布` 是资源
- `flow tab` 是这个资源的固定入口
- 二者不是两个独立领域对象

## 5. ProjectOutline 的定位

`ProjectOutline` 继续保持“派生视图”定位：

1. 输入：项目设置 + 资源文本
2. 输出：导航节点树
3. 用途：浏览、定位、结构感知

它不负责：

1. 直接编辑资源内容
2. 直接承担流程画布职责
3. 代替协作文档或画布

## 6. 当前文档分层

当前仓库文档分成四类：

1. `README.md`
   - 当前产品说明
2. `plan.md`
   - 当前阶段重构计划
3. `docs/*.md`
   - 当前态模型文档与操作说明
4. `docs/archive/*.md`
   - 历史方案归档

## 7. 一句话结论

当前信息架构的核心约束是：

> 所有项目内资料都先收敛为 `ProjectResource`，再按 `resourceKind + collabPurpose` 决定它在页面中的入口、名称和交互方式。
