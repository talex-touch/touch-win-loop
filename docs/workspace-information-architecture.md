# 工作台信息架构

更新时间：2026-04-17

本文描述当前 `touch-win-loop` 已落地的工作台对象关系，目标是让产品文档、接口设计和页面组织围绕同一套模型表达。

## 1. 核心对象

### Workspace

- 个人或 Team 级空间。
- 负责成员、席位、邀请、权限和 AI 配额边界。

### Project

- 工作推进主对象。
- 挂载项目设置、竞赛绑定、资源池、协作状态、AI 提案和 Issue。

### ProjectResource

- 项目内统一的一等资料对象。
- 文件、协作文档、画布都以资源形式存在。

### ProjectOutline

- 由项目设置和资源内容自动推导出的结构化导航。
- 用于浏览和定位，不承担编辑职责。

## 2. 当前页面对应关系

### 左侧资源管理器

- 资源列表
- 项目大纲
- 系统资料库导入
- 上传入口
- 协作资源新建入口

### 主工作区固定页签

- `仪表盘`
- `项目协作`
- `流程画布`
- `项目设置`

### 工作台模式

- `project`
  - 默认研发工作台
  - 保留左侧资源模块、中区 tabs 和右侧 AI dock
- `defense`
  - 独立答辩工作台
  - 左侧为答辩概述与评委列表，中区为比赛状态驾驶舱，右侧为 `AgentDef`
- `final_review`
  - 独立终审工作台
  - 保留终审壳、资料抽屉和助手抽屉，但不再复用研发主区 tabs

答辩工作台当前落地进度、已完成项和未完成项详见：

- [答辩工作台改造进度](./workspace-defense-workbench-progress.md)

### 资源页签

- 某条资源被打开后，会以 `resource:<id>` 形式进入可关闭资源 tab
- `markdown` 协作文档在资源页签内支持 AI 上下文补齐，编辑器会自动预测后续文本，用户按 `Tab` 接受
- 每次接受补齐建议暂按 `0.1 credits` 计入工作空间 AI 配额消耗

## 3. 资源模型

### 资源形态

| resourceKind | 说明                     |
| ------------ | ------------------------ |
| `binary`     | 上传文件或系统资料库引用 |
| `markdown`   | 结构化协作文档           |
| `draw`       | 画布类协作资源           |

### 协作用途

| collabPurpose | 适用形态   | 用户侧名称 |
| ------------- | ---------- | ---------- |
| `workflow`    | `draw`     | 流程画布   |
| `freeform`    | `draw`     | 自由画布   |
| `notes`       | `markdown` | 协作文档   |

## 4. flow tab 的定位

当前约定中，`flow` 固定 tab 不再是旧的静态流程 checklist。

它的职责是：

1. 打开项目唯一的 `workflow` 画布
2. 渲染与资源预览页相同的真实画布协作视图
3. 保持“流程入口”与“workflow 资源”指向同一个底层对象

因此：

- `流程画布` 是资源
- `flow tab` 是这个资源的固定入口
- 二者不是两个独立领域对象

## 5. ProjectOutline 的定位

`ProjectOutline` 继续保持“派生视图”定位：

1. 输入：项目设置 + 资源文本
2. 输出：导航节点树
3. 用途：浏览、定位、结构感知

它不负责：

1. 直接编辑资源内容
2. 直接承担流程画布职责
3. 代替协作文档或画布

## 6. 当前文档分层

当前仓库文档分成四类：

1. `README.md`
   - 当前产品说明
2. `plan.md`
   - 当前阶段重构计划
3. `docs/*.md`
   - 当前态模型文档与操作说明
4. `docs/archive/*.md`
   - 历史方案归档

## 7. 一句话结论

当前信息架构的核心约束是：

> 所有项目内资料都先收敛为 `ProjectResource`，再按 `resourceKind + collabPurpose` 决定它在页面中的入口、名称和交互方式。
